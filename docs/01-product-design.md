## 产品设计与说明文档
**文档版本**：v1.0  
**文档类型**：产品设计说明书（PRD）  
**发布日期**：2026-04-15  
**适用对象**：产品负责人、架构师、研发负责人、AI Agent 平台负责人

---

## 1. 文档目的

本文档用于定义 **OpenClaw Memory Fabric** 产品方案。该方案以 **OpenClaw 插件 + 插件附带 Skills** 的形态接入，在**不修改 OpenClaw、OpenViking、Graphify 源码**的前提下，全面升级 OpenClaw 多 Agent 记忆系统，使其具备：

- 多 Agent 记忆隔离
- 会话跨天持久化
- 长期记忆沉淀
- 复杂项目中的结构认知优先能力
- 更强的自我理解、自我校正与自我回顾能力

---

## 2. 产品背景

当前多数 AI Agent 运行在“短上下文 + 临时工具调用”的模式下，存在如下问题：

1. **多 Agent 记忆污染**  
   不同 Agent 容易混用上下文，导致角色串味、任务边界不清。

2. **会话跨天失忆**  
   当天形成的关键结论、项目背景、用户偏好，次日难以稳定复用。

3. **长期记忆不可治理**  
   大量碎片化记忆进入系统后，很难区分临时信息与长期有效信息。

4. **复杂项目缺少结构认知**  
   Agent 容易直接检索、直接执行，却没有先形成“项目结构地图”，导致检索盲目、执行低效。

5. **缺少自我理解能力**  
   Agent 往往只能“记住信息”，但不知道自己“已经知道什么、还不知道什么、下一步该补什么证据”。

---

## 3. 产品目标

### 3.1 总体目标
建设一个面向 OpenClaw 的企业级多 Agent 记忆系统，使 Agent 从“会调工具”升级为“会建立项目理解、会沉淀长期知识、会在任务中持续进化的智能体”。

### 3.2 核心目标
- 为每个 Agent 提供独立、可治理、可扩展的记忆空间
- 支持项目级长期记忆资产沉淀
- 让 Agent 先建立结构认知，再决定检索与执行
- 支持私有记忆、项目共享记忆、组织共享记忆三层协同
- 支持通过插件与 Skills 双层机制实现“能力 + 纪律”一体化

### 3.3 非目标
以下内容不在本期范围内：
- 不修改 OpenClaw 核心源码
- 不改造 OpenViking 核心检索与存储机制
- 不修改 Graphify 图谱生成核心逻辑
- 不实现跨组织的记忆联邦同步
- 不将本方案做成通用 SaaS 产品

---

## 4. 产品定位

**OpenClaw Memory Fabric** 是一个“记忆编排层产品”，而不是新的 Agent 框架。

它的定位如下：

- **对 OpenClaw 而言**：是增强型记忆与认知插件
- **对 OpenViking 而言**：是上层长期记忆治理器
- **对 Graphify 而言**：是结构认知调用器
- **对 Agent 而言**：是“记忆 + 认知 + 约束”统一工作底座

## 5. 用户与场景

## 5.1 目标用户
1. 企业内部 AI 平台团队
2. 复杂项目的 AI 协作团队
3. 使用 OpenClaw 进行多 Agent 编排的研发团队
4. 需要长期项目沉淀与跨会话协作的产品/研发/运营组织

## 5.2 典型角色
- **主控 Agent**：接收任务、做规划、发起子任务
- **研发 Agent**：处理代码、架构、排障
- **产品 Agent**：处理需求、文档、决策
- **运营 Agent**：处理投放、分析、用户反馈
- **知识整理 Agent**：负责总结、归档、发布共享记忆

## 5.3 核心场景
### 场景 A：复杂项目跨天推进
用户今天让 Agent 分析项目结构、输出方案；明天继续追问时，Agent 能延续昨天的理解，不从零开始。

### 场景 B：多 Agent 并行协作
架构 Agent、实现 Agent、测试 Agent 同时参与一个项目，但各自有私有记忆空间，只有被批准的信息才同步到项目共享区。

### 场景 C：复杂问题前先建结构认知
面对大型代码仓、PRD、会议纪要、历史方案，Agent 先读项目图谱摘要，再决定查什么、做什么。

### 场景 D：长期沉淀组织知识
多次任务中沉淀出的决策、模式、经验，逐步变成可复用的项目资产与共享能力。

---

## 6. 产品原则

### 6.1 隔离优先
不同 Agent 默认不共享私有记忆，避免上下文污染。

### 6.2 结构优先
面对复杂问题，优先生成结构认知，不允许一上来盲检索、盲执行。

### 6.3 长期有效优先
长期记忆只保留稳定事实、偏好、决策、模式，不收纳短期噪音。

### 6.4 自我认知优先
系统不只保存“内容”，还保存 Agent 对自己理解状态的认知。

### 6.5 插件原生优先
基础设施尽量放在 OpenClaw 插件层；流程纪律放在 Skills 层；重计算放到 Sidecar 层。

## 8. 关键产品概念

## 8.1 记忆作用域（Memory Scope）
系统定义三层记忆作用域：

- **Agent Private**：仅当前 Agent 可读写
- **Project Shared**：项目内允许共享
- **Org Shared**：组织范围共享

## 8.2 稳定记忆载体（Stable Carriers）
为了避免记忆碎片化，本系统引入固定命名的长期载体文件，如：

- `identity.md`
- `working-style.md`
- `project-model.md`
- `decision-log.md`
- `entities-glossary.md`
- `playbooks.md`
- `open-questions.md`
- `execution-journal.md`
- `self-model.md`

## 8.3 结构认知摘要（Structural Brief）
由 Graphify 图谱结果生成的项目结构摘要，供 Agent 在任务前阅读。包含：
- 核心对象
- 关键模块/社区
- 主要关系链
- 重要决策点
- 未知点
- 建议检索路径

## 8.4 自我模型卡（Self-Model Card）
记录 Agent 对当前项目的自我理解状态，而非仅记录客观事实。包括：
- 当前目标
- 已理解部分
- 不确定部分
- 缺失证据
- 建议下一步动作

## 9. 功能需求

## 9.1 多 Agent 记忆隔离
### 需求说明
系统必须支持为不同 Agent 分配独立记忆空间，默认情况下互不可读写。

### 规则
- 每个 Agent 拥有私有记忆目录
- 每个项目拥有项目级共享目录
- 组织共享目录仅允许显式发布
- 不允许私有记忆自动外泄到共享区

### 验收标准
- A Agent 写入的私有记忆，B Agent 默认不可见
- 项目共享区内容，可被授权的多个 Agent 检索
- 共享内容可追踪来源与发布时间

## 9.2 跨天持久化
### 需求说明
Agent 隔天启动后，应保留昨日及更早的重要上下文，不从零开始。

### 规则
- 每轮交互后自动蒸馏关键内容
- 持久化写入 OpenViking 和记忆载体文件
- 新会话开始时自动进行召回注入

### 验收标准
- 关闭后重启 OpenClaw，Agent 仍能正确回答前日项目背景
- 关键决策、术语表、工作偏好可稳定召回

## 9.3 长期记忆沉淀
### 需求说明
系统能将高价值信息转化为长期资产，而不是简单堆积聊天记录。

### 规则
- 仅长期有效内容写入稳定载体
- 临时猜测不进入长期记忆
- 争议信息写入 `open-questions.md`

### 验收标准
- 多轮任务后能看到持续增长的项目知识资产
- 冲突信息可追踪，不被静默覆盖

## 9.4 结构认知优先
### 需求说明
复杂任务中，Agent 需要先理解项目结构，再做检索与执行。

### 触发条件
- 首次接触新项目
- 项目资料量大
- 用户问题跨模块
- Agent 当前结构理解置信度低

### 验收标准
- Agent 能先输出“当前结构理解”
- 后续检索动作与该结构理解一致
- 无结构摘要时，会主动触发图谱引导流程

## 9.5 自我理解能力
### 需求说明
系统要支持 Agent 持续维护“我知道什么/我不知道什么”的状态。

### 验收标准
- 每个项目存在 `self-model.md`
- Agent 可陈述自己当前的理解边界
- 对复杂任务能给出“还需要哪些证据”的判断

## 9.6 共享记忆治理
### 需求说明
支持从私有记忆中挑选高价值内容，发布到项目或组织共享区。

### 规则
- 必须通过显式发布工具完成
- 发布内容带来源、时间、发布者、可见性标签
- 支持后续撤回或废弃

---

## 10. 非功能需求

## 10.1 性能要求
- 单次记忆简报生成：P95 ≤ 2 秒（本地中等规模项目）
- 单次项目结构摘要读取：P95 ≤ 3 秒
- Graphify 深度重建为异步/离线触发，不阻塞正常问答

## 10.2 稳定性要求
- 插件故障不应破坏 OpenClaw 基础可用性
- 图谱服务不可用时，应降级到 OpenViking + 本地记忆载体
- OpenViking 暂时不可用时，应至少使用本地稳定载体

## 10.3 安全性要求
- 私有记忆默认不共享
- 发布共享必须经过显式动作
- 不保存未授权敏感内容到共享域
- 保留操作审计日志

## 10.4 可观测性要求
- 记录 recall 来源、耗时、命中范围
- 记录 commit 写入项、作用域、冲突数
- 记录图谱是否过期、是否触发更新
- 提供健康检查命令与状态摘要

## 12. 用户流程

## 12.1 新项目接入流程
1. 用户选择项目目录或知识资料目录
2. 插件初始化项目作用域
3. Graphify 生成首版图谱
4. OpenViking 建立项目记忆命名空间
5. 初始化 `project-model.md` 与 `self-model.md`
6. Agent 开始基于结构认知回答问题

## 12.2 日常任务流程
1. 用户发起任务
2. 插件识别 Agent 与项目上下文
3. 自动生成 Memory Brief
4. 若为复杂任务，则优先读取 Structural Brief
5. Agent 决定检索 / 执行 / 分派子任务
6. 任务结束后自动蒸馏与回写

## 12.3 共享发布流程
1. Agent 或用户发起发布动作
2. 选择要共享的记忆条目
3. 标注可见性与说明
4. 写入共享目录
5. 其他 Agent 在相关项目中可召回

## 14. 风险与对策

| 风险 | 说明 | 对策 |
|---|---|---|
| 记忆过载 | 召回过多导致上下文膨胀 | 采用 L0/L1/L2 分层与预算控制 |
| 记忆污染 | 临时信息进入长期载体 | 引入蒸馏规则和冲突隔离 |
| 图谱过期 | 项目变化后结构认知失真 | 支持手动/自动增量更新 |
| 插件过重 | 插件内做太多计算影响网关 | 重计算放到 sidecar |
| 共享泄漏 | 私有记忆误入共享区 | 默认 private，显式 publish |
| 依赖不可用 | OpenViking/Graphify 某一层异常 | 多级降级策略 |

## 16. 验收标准

满足以下条件视为一期验收通过：

1. 至少支持 3 个 Agent 的私有记忆隔离
2. 跨天任务可延续上下文
3. 项目级长期载体可持续增长且结构清晰
4. 复杂任务可先输出结构理解，再进行检索执行
5. 可通过显式工具发布共享记忆
6. 部署完成后具备健康检查与日志观测能力
7. 全程不修改 OpenClaw、OpenViking、Graphify 源码

---

## 17. 结论

OpenClaw Memory Fabric 的核心价值，不是“多存一点记忆”，而是把 OpenClaw 的 Agent 从“短期会话工具使用者”升级为“有边界、有结构、有沉淀、会自我理解的长期协作体”。

它将成为 OpenClaw 上层最关键的记忆与认知基础设施，为多 Agent 长期复杂协作提供可持续底座。
